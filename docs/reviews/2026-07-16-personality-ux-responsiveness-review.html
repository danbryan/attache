Responsiveness: what is slow, how to measure it, how to fix it

What I found

FindingEvidence
No hard hangsmacOS writes .hang/.spin reports to DiagnosticReports when an app stalls; none exist for Attaché. The "little slow" feel is sub-second render latency, not a beachball.
God-object observationContextSettingsPane declares @ObservedObject var model: AppModel. AppModel publishes at high frequency during narration, character animation, and playback, so the whole Settings pane re-renders on every tick even though nothing it shows changed.
Per-render recomputationactiveCapabilitySummary (ContextStrategyViews.swift:420) rebuilds the capability profile and its formatted labels on every body evaluation. The cache read is in-memory (cheap), but the rebuild plus label formatting runs at the re-render frequency above, and the Advanced DisclosureGroup animates its expansion while that churn is happening.
Voice catalog blocks the main threadFound during the personality review below: system-voice enumeration runs synchronously on the main thread during app init, and the code's own comment says it can take seconds. This is a launch-time stall rather than an Advanced-pane one, but it is the largest single main-thread block in the app.
No instrumentation existsZero os_signpost, MetricKit, or timing probes anywhere in Sources. Today the only responsiveness signal is human perception, which is why "maybe it was just my Mac" is unanswerable.

How to measure (build this first)

  1. Main-thread watchdog: a background queue pings the main runloop every 100ms and records any stall over 250ms with the frontmost pane name. Content-free, always on, surfaces in Diagnostics. About a day of work, answers "was it the app or my Mac" forever.
  2. Signpost intervals around the expensive moments: settings pane switch, Advanced expansion, palette open, personality switch, first caption render. Instruments' Hangs and Time Profiler then attribute every slow interval to real code.
  3. A latency gate in ui-smoke: the AX harness already opens Settings and palettes; assert each opens under a budget (for example 300ms) so regressions fail CI instead of reaching your hands.

How to fix (after measuring)

  1. Scope observation. Give Settings panes a small focused observable (strategy, capability summary, provider label) instead of all of AppModel. The character renderer keeps its high-frequency feed; Settings stops re-rendering per animation frame.
  2. Compute capability summaries on change, not on render. Recompute when provider, model, base URL, or override changes; store the formatted summary; body just reads it.
  3. Keep the Advanced expansion animation light. Once the two fixes above land, the DisclosureGroup animates against a quiet pane and should feel instant.

Recap: the unbounded-input problem is real

Verified The earlier warning was correct and still stands, with one improvement since: the recap request now compiles through the context compiler, so a huge recap can no longer silently ship an oversized request to a provider. But the assembly layer (AppModel.swift:2390) maps every unread card into the prompt with no item cap, and because the items are baked into the current user turn they count as protected content that must always fit. The dynamic sentence ceiling bounds the output, not the input.

Failure mode today: recap over a very large inbox hits protected-content overflow and errors out. Nothing leaks and nothing silently truncates, which is correct, but the user asked for a recap and gets an error instead of a recap.

Recommended fix: stage it like whole-session review

  • Move recap items out of the protected user turn and into context items with the summarizeEligible treatment, so the compiler can compress or stage them instead of failing.
  • Cluster cards by session (the recap prompt already clusters conceptually), cap each stage to what the budget planner allows, and roll stage summaries into the final spoken recap. The staged machinery from whole-session review (INF-329) is the pattern to reuse.
  • Above a threshold (say 40 items), show the same cost-preview affordance whole-session review has: "Recapping 214 updates across 12 sessions will take N model calls."

Private call: keep fail-closed, add two entry points

Your instinct is right that the current flow is awkward: privacy is a decision you have to make before the call exists, at the moment you are least thinking about it. Both of your ideas work, and they compose:

ProposalDesignAssessment
Right-click Call → Private CallContext menu on the Call control with "Start Private Call". Keeps the storage decision frozen at call start, which the current architecture already guarantees (the storage mode is captured when the call begins and never mutates).Build first. Cheap, discoverable, zero semantic risk, consistent with the right-click direction below.
Switch to private mid-call"Make this call private" action during a saved call. Semantics must be retroactive: hard-delete every already-persisted card, capsule, and reply for this conversation id, then continue in memory only. The pieces exist: saved calls carry a conversation id, and the store now has secure-delete with WAL truncation. If the retroactive scrub fails, the call stays saved and says so; never claim private when storage still holds turns.Build second. More work and needs the fail-closed rule, but it matches how people actually realize a call should be off the record: partway through.
One-way door: allow saved→private, never private→saved. Retroactively persisting turns the user believed were ephemeral is the worst possible surprise. If they want a record, they can start a new saved call.

Whichever entry is used, show a persistent, unmistakable PRIVATE badge in the call HUD for the whole call. Privacy states users cannot see get distrusted.

The character, playback, and what to add next

The sprite is a moat, not a gimmick

Agreed on the novelty. Nobody else in this category has a presence that watches sessions, follows dragged cards, and reacts to activity. It converts an infrastructure product into something people show their friends, and it is the single most screenshot-able thing in the app.

  • Guard it with Reduce Motion. Honor the system accessibility setting with a calm variant. Delight that ignores accessibility settings reads as neglect.
  • Keep it out of the data's way. The character should yield (dim, step aside) whenever captions or cards are the user's focus.
  • Its animation feed is the same publish churn hitting Settings. Fixing scoped observation makes the sprite cheaper too.

More of what already works

  • Drag a session card onto the character to focus that session. The physicality matches the pet interaction you already have.
  • Number keys 1 through 9 switch to that personality while the ⇧⌘P palette is open.
  • j / k navigation in the ⌘I inbox and ⌘Y history, matching the type-to-filter that exists.
  • Hover-scrub captions: hovering a karaoke word previews position, click jumps (click already works).
  • Global summon hotkey (user-configurable) to bring the mini window or dock forward from any app, since Attaché is a background utility by design.

Supporting more harnesses and open-source models

Good news: the architecture is closer to pluggable than you might think

Two clean protocols already exist: SessionScanner (discovery and parsing) and InstructionDeliveryAdapter (two-way replies). Once a transcript is parsed, everything downstream is source-agnostic: FTS indexing, cards, captions, playback, memory, and personalities all carry the source as an opaque string. The hardcoded seams are the TranscriptFormat enum (currently exactly claude and codex), the live watcher's directory list and source classifier, and the two-way readiness and correlation scanners.

TierWhat a harness getsWhat it takes
Tier 0: event POST works todayNarrated voicemail cards with a sensible badge, playback, recap inclusion. No live transcript tail, no activity phrases, no two-way.Zero app changes. The local event server accepts any source string on POST /events with the per-launch token. The missing piece is a documented public event schema and an "integrate your harness" guide; today the JSON shape is only a de facto contract.
Tier 1: watched and indexedEverything above plus live narration, session discovery and search, activity phrases, the character reacting to the session.Roughly five required files per harness: a SourceKind case, a TranscriptFormat case, a paths helper, a SessionScanner implementation, and a parser branch; plus extending the two watchers from binary claude-or-codex classification to a multi-source registry. That watcher generalization is the one-time structural investment that makes the third, fourth, fifth harness cheap.
Tier 2: full two-wayReply into the session from a call, with the existing frozen-target safety model.Readiness classifier, reply correlation, and a delivery adapter per harness. Hard feasibility gate: the harness must offer a scriptable, non-interactive "resume session by id" that writes to the same transcript and emits verifiable completion evidence. Claude Code and Codex have this; candidates vary widely here.
Recommended sequence: first publish the Tier 0 event schema (a docs page plus a stable versioned JSON contract; near-zero code), then do the watcher generalization once, then add the first new harness at Tier 1 to prove the abstraction. Pick the first harness by measurable demand: Gemini CLI and opencode both keep on-disk session files and are the strongest Tier 1 candidates; Cline and Roo Code live inside VS Code and fit Tier 0 best initially. Promote a harness to Tier 2 only after its resume primitive is verified against docs/two-way.md's safety rules.

Open-source models: already a solved path

Verified The personality (narration) model is fully decoupled from which harnesses are watched, and Ollama is a first-class provider: model discovery, capability detection through /api/show (context length, reasoning support), correct on-device egress classification, no API key, and canary scripts proving tool calling. Groq and any OpenAI-compatible endpoint are also in. Anyone running open-source models locally can use Attaché fully today; sessions from open-source harnesses are the Tier 1 work above.

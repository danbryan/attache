Memory, MCP, and connectors for Attaché

Attaché already has a careful local memory system and a provider-neutral tool loop. The best next move is to make Attaché an MCP host with gateway and direct-server support, user-selected approval modes, and Ask Attaché versus Tell Agent as the hard routing boundary.

Memory should be global and personality-specific

Global user memory

Use for stable facts and preferences that should follow the person across every character.

  • Preferred name, timezone, pronouns, and recurring preferences
  • Stable household or relationship facts the user clearly shared
  • Cross-character standing instructions such as answer length

Personality memory

Use for a relationship or shared history whose meaning depends on one character.

  • “Colt calls me boss” or an inside joke with Colt
  • A tone preference that applies only to Attaché
  • A character-specific ritual, boundary, or ongoing thread

The storage model is already right

Verified AttacheMemoryScope already has global, personality(id), and topic(key). The selector filters scope before text becomes model context. Switching personalities does not copy or leak character-scoped records.

Recommended UX adjustment: keep ordinary facts global by default, but put an explicit scope selector on suggestion review and edit: “All personalities,” “Only Colt,” and, later, “This topic.” Show the active character’s name instead of the vague label “This personality.”

Do not make memory itself a transcript archive. Store compact durable claims with provenance, confidence, timestamps, and supersession. Keep full chat and agent sessions in their existing stores, then retrieve bounded evidence when a question needs it. LongMemEval specifically evaluates extraction, multi-session reasoning, temporal reasoning, updates, and abstention, and reports that long context alone still degrades materially on sustained histories.

Attaché’s existing mix of structured facts, on-demand retrieval, rolling summaries, and context budgets follows the useful part of both hierarchical-memory research and long-term assistant benchmarks without requiring a vector database.

MetaMCP and direct servers should coexist

A gateway is not a separate integration type from Attaché’s perspective. A MetaMCP route is a normal Streamable HTTP MCP connection whose upstream servers, authentication, and filtering happen behind the endpoint. Direct hosted and local servers use the same registry but let Attaché manage their lifecycle and authentication itself.

Separate connection from access

Connection registry: app-wide transport and credentials. Tool profile: named collection of servers and tool filters. Personality grant: a tool profile plus approval mode. Several personalities can reuse one connection without sharing character memory or duplicating secrets.

Do not load a gateway’s whole catalog

Gateways simplify management but can still expose hundreds of schemas. Attaché should index names and descriptions locally, search the catalog, inspect only likely tools, and then execute. The official MCP client guidance recommends this catalog, inspect, execute pattern once schemas consume roughly 1 to 5 percent of the context window.

Recommended personality setup: a Tools section chooses “No tools,” an app-default profile, or a named profile. A MetaMCP user can assign one route per character. A direct-MCP user can build the same profile by selecting several individual servers. The model sees the resulting capability set, not how it was assembled.

Routing should remain simple and visible

The destination control answers “who handles this turn?” Tool selection happens only inside the chosen destination.

Ask Attaché
The personality model sees enabled, relevant tools. It can choose a direct read without asking permission every time if policy marks that exact tool safe. Writes and sensitive reads can pause for native approval.
Tell Agent
The raw turn goes to the frozen Codex or Claude Code session. Attaché does not also call MCP, reinterpret the request, or silently switch back to its own tools.
Ask Attaché + “ask Codex”
The existing stage_agent_instruction tool opens native confirmation because the user explicitly requested a handoff.
No matching tool
Attaché says the needed integration is unavailable and can offer to open Integrations. It does not treat Codex or Claude Code as an automatic fallback destination.

Safety and context implications

User-selectable approval mode

  • Full access: run every tool in the personality’s selected profile without per-call prompts.
  • Automatic: run locally trusted reads automatically and ask for writes, destructive calls, sensitive reads, and open-world actions.
  • Ask every time: show the server, tool, frozen arguments, and destination before every call.
  • Read only: expose only locally verified read tools, useful for research and planning.
  • Tools off: expose no MCP tools to the personality.

Context policy

  • Discover tools progressively. Do not inject every schema from every server into every turn.
  • Bound and classify tool results before they enter model context.
  • Mark external results as untrusted evidence, not instructions.
  • Add server, tool, argument category, result size, disposition, and approval to the context receipt without logging sensitive content.

Credential boundary

  • Store remote OAuth and API credentials in Keychain.
  • Keep stdio process environments minimal and explicit.
  • Pin server identity and display the actual remote host.
  • Never make a personality export include credentials or account grants.

Cross-server flow

  • A result from server A is untrusted input to server B.
  • Do not allow an unapproved chain to move mail, files, or secrets between services.
  • Carry the current effect ledger across retries and fallbacks.
  • Never replay a write automatically after an uncertain timeout.
Fast switching: show the active approval mode in the main window and let Tab cycle forward or Shift-Tab cycle backward only when no text field, menu, sheet, or modal control owns focus. Freeze the selected mode for each turn so a mode change cannot alter an already proposed call.
The MCP specification exposes readOnlyHint, destructiveHint, idempotentHint, and openWorldHint, but explicitly treats annotations as untrusted hints. Attaché should map them into a local classification. A trusted gateway can supply overrides, but the user’s Attaché policy remains the final authority.

Recommended implementation sequence

PHASE 0

Contracts and threat model

Define server records, tool identity, local risk overrides, per-personality grants, credential references, execution receipts, output caps, timeout semantics, and the no-double-route rule. Add adversarial tests before network execution.

PHASE 1

Connection registry and gateway canary

Support remote Streamable HTTP and local stdio, Keychain credential references, OAuth for hosted servers, MetaMCP-style bearer endpoints, and unlimited named connections. Prove discovery and a read call against both one gateway route and one direct server.

PHASE 2

Tool profiles and approval modes

Add named tool profiles, per-personality assignment, Full access, Automatic, Ask every time, Read only, and Tools off. Use one consistent confirmation surface with frozen arguments, effect deduplication, non-replayable fallbacks, audit records, and clear partial-failure states.

PHASE 3

Progressive discovery and resources

Index server and tool names locally, offer only the small relevant tool set to the model, and add MCP resources only when they provide a meaningful advantage over tools. Keep prompts disabled until their trust model is designed.

PHASE 4

Optional provider connectors

Add OpenAI connectors only as a convenience path for eligible OpenAI Responses models. Label that path clearly because it changes the data route and does not generalize to local or other hosted models.

Dependency decision

The official MCP Swift SDK supports clients and macOS 13+, which matches Attaché’s platform floor. It is pre-1.0 and currently requires evaluating Swift toolchain compatibility and pinning strategy. Attaché presently has only Sparkle as an external dependency, so this deserves a deliberate architecture ticket and a small protocol spike.

Ownership model

Server connection and credential: app-wide. Tool profiles: app-wide reusable collections. Profile and approval mode: per personality, with an optional app default. Memory: global, personality, or topic scoped. Agent send authority: always frozen per focused session and separate from MCP grants.
